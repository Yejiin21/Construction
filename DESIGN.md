## 데이터 분석

**처음엔 단순한 3계층 구조라고 판단했지만, 전수 분석을 통해 케이스마다 스키마 자체가 달라지는 구조임을 이해했습니다.**

처음에는 `drawing → discipline → revision`의 3계층 구조라고 생각했습니다. 실제로 데이터를 끝까지 읽고 나서야 세 가지 예외 케이스가 있음을 발견했습니다.

101동 구조는 revisions 대신 regions: { A, B }가 있고 각 region이 독립적인 revision을 가졌습니다. 주민공동시설 건축은 discipline level에 imageTransform과 polygon이 없고 revision마다 각자 보유하고 있었습니다. 주차장 구조는 polygon 자체가 없었습니다.

처음에는 데이터 오류라고 생각했는데, 도면 사진을 보며 생각해보니 전부 현장 상황을 반영한 결과였습니다.
구조 도면의 A/B 분리는 시공팀이 실제로 나뉘기 때문이고, 주차장 구조에 polygon이 없는 건 그 공증 자체가 기준 도면이기 때문이었습니다.
이 예외들이 도메인의 복잡성을 그대로 담고 있다는 걸 이해하는 것이 설계에서 중요하다고 생각합니다.

파악 과정에서 가장 중요했던 건 imageTransform과 polygonTransform의 역할 구분이었습니다.
둘 다 x, y, scale, rotation 구조를 공유해서 혼동하기 쉬운데, imageTransform은 이미지를 다른 이미지 위에 CSS transform으로 합성하는 변환이고, polygonTransform은 polygon 꼭짓점 좌표를 화면 좌표계로 바꾸는 수학 변환입니다. 이 구분 없이 렌더링 로직을 짜면 구조 전체가 잘못됩니다.

**데이터를 직접 다루면서 불편했던 지점 세 가지가 개선 포인트라고 생각합니다.**

첫 번째로는 relativeTo를 파일명 대신 drawing.id로 참조해야 합니다.
"relativeTo": "01*101동 지상1층 평면도*건축.png" 형태인데, 파일명이 바뀌는 순간 이를 참조하는 모든 공종의 합성 변환이 깨집니다. drawing에 이미 id 필드가 있으므로 id 참조하는 것으로 개선할 수 있습니다.

두 번째, base 공증을 데이터에 명시해야 합니다.
현재는 런타임에 "다른 공종들의 relativeTo가 가리키는 이미지를 가진 공종"을 역추적해서 base를 찾습니다. "isBase": true 하나만 추가해도 이 역추적 로직이 불필요해짐으로써 개선할 수 있습니다.

세 번째, imageTransform과 polygon의 위치를 통일해야 합니다.
표준 케이스는 discipline level에 있고, 주민공동시설 건축은 revision level에 있습니다. 이 비일관성 때문에 코드에서 "discipline level에 없으면 revision 것을 쓴다"는 분기가 계속 생겼습니다.
revision을 항상 최종 접근 경로로 고정하고 공통 값은 discipline에 default로 두는 방식으로 개선되면 이런 분기가 감소할 수 있습니다.

## 접근 방식

**데이터 구조를 완전히 파악한 이후에 설계를 시작하는 순서로 진행했습니다.**

데이터를 파악 하지 않을 채 UI를 설계하면, 나중에 예외 케이스를 발견했을 때 구조를 뒤집어야 하는 상황이 생길 수 있다고 생각했습니다. 그래서 진행 순서를 다음과 같이 진행했습니다.

가장 먼저 metadata.json의 특수 케이스 세 가지를 먼저 파악했습니다.
이 단계에서 imageTransform과 polygonTransform의 역할 차이, relativeTo 방향성, region 구조의 존재를 확인했습니다. 데이터 파악을 먼저 한 덕분에 region 케이스를 설계 이후에 발견해 코드를 대거 수정하는 상황을 피할 수 있었습니다.

두번째로 탐색 계층(drawing → discipline → region → revision)과 오버레이 상태가 어떻게 연결되는지를 먼저 Zustand store로 설계했습니다.
selectDrawing()을 호출하면 하위 상태(discipline, region, revision, overlay 목록)가 모두 초기화되는 도메인 규칙을 store action에 응집시켰습니다. 이렇게 하면 컴포넌트는 selectDrawing(id) 한 줄만 호출하면 되고, 초기화 순서를 알 필요가 없습니다.

세번째로 공간 중심 트리 사이드바 + 우측 캔버스 + 하단 리비전 바 구조를 선택했습니다.
탭이나 공종 중심 트리도 검토했으나, 사용자가 "101동 건축 도면을 보러 간다"처럼 공간 기준으로 탐색한다는 점에서 공간 중심 트리가 더 자연스럽다고 판단했습니다. 리비전은 사이드바 대신 하단 바에 가로로 배치해 시간 흐름 방향을 직관적으로 맞추고, 캔버스 영역도 온전히 확보했습니다.

마지막으로는 imageTransform의 x, y, scale, rotation을 CSS transform에 직접 매핑하며 렌더링을 구현했습니다.
외부 라이브러리 없이도 충분히 구현 가능한 수준이었고, 불필요한 의존성을 추가하는 것보다 직접 구현이 낫다고 판단했습니다.

구현 과정에서 가장 예상치 못한 문제는 Region A/B 오버레이였습니다.
imageTransform을 그대로 적용하면 이론상 위치가 나와야 하는데, 실제로는 리비전마다 scale·rotation 값이 달라 도면 영역을 크게 벗어나거나 비율이 어긋나는 경우가 있었습니다. 메타데이터만으로는 사용자가 원하는 비교 화면을 만들기 어렵다고 판단해, 사용자가 위치·크기·회전을 직접 보정할 수 있는 캘리브레이션을 추가로 도입했습니다. 캘리브레이션 값은 메타데이터 transform에 덮어쓰지 않고 delta로 별도 저장해, 원본 데이터와 사용자 조정값을 명확히 분리했습니다.

## UI 설계 결정

### 레이아웃 — wiki 스타일 트리 사이드바

**공간 중심 트리 사이드바 + 우측 캔버스 + 하단 리비전 바 구조를 선택했습니다.**

고려했던 대안과 각각의 판단은 다음과 같습니다.

첫 번째 대안은 탭 형태였습니다. 동/공종/리비전을 각각 탭으로 나누는 방식입니다. 화면 공간을 효율적으로 쓸 수 있다는 장점이 있지만, 탭을 이동할 때마다 현재 어디를 보고 있는지 컨텍스트를 잃게 됩니다. 현장 소장 입장에서 "지금 101동 건축 REV1을 보고 있다"는 위치 인식이 중요한데, 탭 방식은 그걸 깨뜨립니다.

두 번째 대안은 공종 중심 트리였습니다. 건축/구조/설비를 최상위로 두고 각 공종 아래 해당 건물들을 나열하는 방식입니다. 공종별 현황을 한눈에 파악하기 좋지만, 사용자 시나리오를 보면 김철수 씨는 "101동 건축 도면을 확인하러 간다"처럼 공간 기준으로 접근합니다. 공종 중심 트리는 그 탐색 흐름과 맞지 않습니다.

최종적으로 공간 중심 트리를 선택했습니다. 데이터 계층(`단지 → 동 → 공종 → region → revision`)을 그대로 트리로 시각화하면 현재 위치가 자연스럽게 드러나고, 레퍼런스로 정한 wiki처럼 익숙한 탐색 경험을 제공합니다.

리비전 타임라인을 하단 바에 배치한 이유는 사이드바에 리비전까지 넣으면 공종 수가 많을 때 스크롤이 복잡해집니다. 하단에 가로로 배치하면 `초기 → REV1 → REV2`의 시간 흐름이 방향적으로 일치하고, 캔버스 전체 영역도 온전히 확보할 수 있습니다.

## 기술 선택

### 상태 관리 — Zustand

**전역 탐색 상태 관리에 Zustand를 선택했습니다.**

탐색 상태와 오버레이 상태가 여러 컴포넌트에 걸쳐 공유되므로 전역 상태 관리가 필요했습니다.

React Context API도 검토했습니다. 하지만 Context는 값이 바뀌면 해당 Context를 구독하는 컴포넌트 전체가 리렌더링됩니다. 탐색 상태가 자주 바뀌는 구조에서는 관련 없는 컴포넌트까지 리렌더링되는 문제가 생깁니다. Redux도 검토했는데, action creator와 reducer를 따로 관리해야 하는 구조가 이 규모의 프로젝트에는 과도하다고 판단했습니다.

Zustand를 선택한 핵심 이유는 도메인 규칙을 store action 한 곳에 응집할 수 있다는 점입니다. 실제 drawingStore.ts를 보면 selectDrawing()을 호출하면 모든 값이 한 번에 초기화됩니다. 이 규칙이 store에 있으면 컴포넌트는 selectDrawing(id) 한 줄만 호출하면 되고, 초기화 순서를 알 필요가 없습니다. selectDiscipline()도 마찬가지로 오버레이 모드 중 다른 공종을 선택하면 자동으로 비교 모드가 종료되는 규칙이 store action 안에 담겨 있습니다.

### 이미지 오버레이 — CSS Transform (외부 라이브러리 없음)

**외부 라이브러리 없이 CSS `transform`으로 이미지를 합성했습니다.**

imageTransform의 x, y, scale, rotation 필드가 CSS transform 파라미터에 직접 매핑되기 때문에 별도 라이브러리 없이 구현 가능했습니다. mixBlendMode: 'multiply'를 선택한 이유는 건축 도면이 흰 배경에 선으로 구성된 경우가 많아서, multiply 블렌딩을 쓰면 두 도면이 자연스럽게 겹쳐 보이기 때문입니다. 단순 opacity만 사용하면 위 이미지가 아래 이미지를 덮어버려 비교가 어렵습니다.

Region 오버레이는 구조 full 이미지 기준 변환(t1)과 region 리비전 이미지 기준 변환(t2)을 중첩 컨테이너로 처리했습니다. CSS transform의 중첩 특성을 그대로 활용한 방식입니다.

## 어려웠던 점 및 개선 방안

### Region A/B 캘리브레이션을 도입

**메타데이터 transform만으로는 Region 도면이 base와 정확히 맞지 않아, 사용자가 직접 보정할 수 있는 캘리브레이션을 추가로 구현했습니다.**

101동 구조는 Region A·B로 나뉘며, 각 region 이미지는 base를 잘라 회전·확대한 부분 도면입니다. imageTransform을 그대로 적용하면 이론상 위치가 나와야 하는데, 실제로는 리비전마다 scale·rotation 값이 달라 도면 영역을 벗어나거나 비율이 어긋났습니다. 메타데이터만으로는 사용자가 원하는 비교 화면을 만들기 어렵다는 판단 하에 캘리브레이션을 도입했습니다.

캘리브레이션 값은 메타데이터 transform을 덮어쓰지 않고 delta(x, y, scale, rotation)로 별도 저장했습니다. 원본 정렬 정보와 사용자 조정값의 의미가 분리되어 초기화도 간단합니다.

구현 과정에서 세 가지 문제가 있었습니다.

첫째, region 리비전의 scale이 2~3처럼 큰 값이어서 그대로 적용하면 이미지가 도면 영역을 벗어났습니다.
컨테이너는 1배로 고정하고 이미지는 object-contain으로 처리한 뒤, region 리비전의 scale은 컨테이너에 적용하지 않고 캘리브레이션 scale만 반영하도록 변경했습니다.

둘째, 오버레이 전체에 pointer-events: none이 걸려 있어 Region을 드래그할 수 없었습니다.
이미지를 그리는 visual 레이어(pointer-events 없음)와 같은 위치·transform을 가진 드래그 전용 레이어(pointer-events: auto)를 분리해 해결했습니다. RegionOverlay의 mode prop이 이 구조를 반영합니다.

셋째, 드래그 시 화면 픽셀 이동량을 메타데이터 좌표계로 변환해야 했습니다.
도면이 object-contain으로 스케일되어 있어서 화면 픽셀 delta를 renderScale로 나눠야 올바른 좌표 이동량이 됩니다. 이를 놓치면 드래그 시 오버레이가 너무 느리거나 과하게 움직이는 문제가 발생했습니다.

### 시간이 더 있다면 개선하고 싶은 것

**전체 배치도 polygon 클릭 탐색과 리비전 간 변경 영역 시각적 diff를 추가하고 싶습니다.**

첫 번째는 배치도 위 건물 영역 클릭으로 해당 동을 선택하는 기능입니다. 현재는 사이드바 트리에서만 동을 선택할 수 있는데, 배치도에 이미 각 building의 `position.vertices`가 있으므로 polygon을 클릭 가능하게 만들면 공간 기반 탐색이 더 직관적으로 완성됩니다.

두 번째는 리비전 이동 시 달라진 관심 영역을 하이라이트하는 diff 기능입니다. 현재는 리비전을 바꿔도 어떤 영역이 변경됐는지 시각적으로 알 수 없습니다. `changes` 필드가 텍스트로만 존재하는 지금 구조에서는 완전한 구현이 어렵지만, polygon 영역 단위로 변경 여부를 표시하는 것만으로도 리비전 검토 흐름이 크게 개선될 것입니다.
