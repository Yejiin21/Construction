## 1. 개발 철학이란 (00:00)

### 코드 리뷰에서 들어본 말

> "변수명으로 정보를 전달하는 게 좋습니다"  
> "코드를 간결하게 작성하세요"

### 어떤 코드가 더 잘 짠 코드일까?

```typescript
// 첫 번째: 변수명으로 정보 전달
const userAge = calculateAge(birthDate);
return userAge;

// 두 번째: 간결하고 메모리 절약
return calculateAge(birthDate);
```

**정답이 있을까요?**

> **네, 맞습니다. 그래서 "개발 철학"이라는 말이 나오는 거예요.**

### 개발 철학의 정의

- 코드를 쓰는 **본인만의 기준**
- 정답이 없음 - 각자의 철학에 따라 선택
- 좋은 코드와 나쁜 코드를 **구분**할 수 있게 해줌
- 단순히 클린 코드 책을 따라하는 수준이 아님

> **본인의 개발 철학이 먼저 확립되고, 그 뒤에 철학에 맞는 방법들을 선택해 나가는 겁니다.**

## 2. 저의 개발 철학 (01:04)

> **"다른 사람의 시간을 줄여 주자"**

### 철학에 따른 선택

- 중복되는 코드를 **의도적으로** 작성하기도 함
- 중복 코드가 오히려 **복잡성을 줄여줌**
- 복잡한 코드를 이해하는 것보다 중복 코드가 **추가하기 쉬움**

## 3. 철학 적용 예시 (02:10)

### ❌ 안 좋은 코드

```typescript
// 이 코드를 이해하려면 전체를 다 봐야 함
async function fetchUserInfo() {
  const userData = await fetchUser();
  const myData = await fetchMyData();
  
  if (!userData) throw new Error();
  if (!myData) throw new Error();
  
  // ... 더 많은 로직
}
```

**문제점:**
- 코드 전체를 봐야 이해 가능
- 가독성이 좋지 않음
- `throw`를 왜 해주는지 이해하는데 시간 소요
- `if`문 순서가 중요한지 파악 필요

> **5명이 이 코드를 보면 5분, 10명이면 10분**

### ✅ 개선된 코드

```typescript
// 읽는 사람에게 선택권을 넘김
async function fetchUserInfo() {
  const { userData, myData } = await loadUserData();
  
  // 가드문 - 이하가 실행 안 될 수도 있겠구나
  if (!UserInfoValidator.validate(userData, myData)) return;
  
  // ... 핵심 로직
}
```

### 싱글톤 패턴 사용 이유

```typescript
// ✅ 틈을 주지 않기 위해 싱글톤 사용
const UserInfoValidator = {
  validate(userData, myData) {
    // ...
  }
};

// ❌ 틈이 생기는 코드
const validations = new UserInfoValidations();
// 여기에 다른 코드가 들어갈 수 있는 틈!
validations.validate(userData, myData);
```

> **동작을 하는 코드의 물리적인 거리가 있으면 다른 내용이라는 착각이 듦**

### 클래스 대신 객체 리터럴

```typescript
// ❌ 클래스 - 불필요한 코드, 싱글톤인지 알기 어려움
class UserInfoValidator {
  private static instance: UserInfoValidator;
  static getInstance() { ... }
  validate() { ... }
}

// ✅ 객체 리터럴 - 단번에 싱글톤임을 알 수 있음
const UserInfoValidator = {
  validate() { ... }
};
```

### if-else로 2분법 명시

```typescript
// ✅ 딱 2가지 상황만 있다고 명시
if (isValid) return true;
else return false;

// 인간의 뇌는 2분법적 사고를 좋아함
// 음과 양, 선과 악, 흑과 백
```

> **"이 코드는 건드릴 필요가 없다"를 우회적으로 표현**

### 중괄호 제거로 확장 방지

```typescript
// ✅ 의도적으로 틈 제거 - 확장하지 않겠다는 의지
if (isValid) return true;
else return false;

// ❌ 중괄호 있으면 다른 코드가 들어갈 틈이 생김
if (isValid) {
  // 여기에 코드 추가 가능...
  return true;
}
```

> **확장성이 좋은 코드도 좋지만, 그렇게 되면 코드를 다 이해해야 함.  
> 단언적인 코드가 있다면 빠르게 넘어갈 수 있음.**

### 느낌표(!) 제거

```typescript
// ❌ 느낌표 - 생각을 한번 더 해야 함
if (!isInvalid) { ... }

// ✅ 느낌표 제거 - 바로 이해
if (isValid) { ... }
```

## 4. 개발 철학을 얻는 방법 (05:20)

> **끊임없이 실패를 해보고 실패를 극복해 보는 것**

### 개발자의 실패

> "코딩을 하다가 코드가 내 손에서 떠나가는 경험"
> "너무 복잡해져서 더 이상 손을 못 댈 것 같은 상황"

### 1만 시간의 법칙

- 단순히 1만 시간을 채우는 게 아님
- **조금씩 단계를 높여가며 연습**

### 실천 방법

1. **간단한 투두 리스트** 만들기
2. **하나씩 기능 추가**해 나가기
3. **"더 이상 기능 추가 못하겠다"** 순간이 옴
4. **코드 분석** - 어디가 복잡한지, 왜 복잡해졌는지
5. **기준 세우기**
   - 어느 정도까지 복잡함을 용인할 수 있는지
   - 비슷한 문제를 겪지 않으려면 무엇이 필요한지
6. **기준에 따라 리팩토링** 시도
7. **실패하면** → 다른 사람 코드 중 기준에 맞는 부분 따라하기
8. **클린 코드 같은 책** 읽기
9. **기준 수정** 필요시 변경
10. **반복**

> **점점 견고해지는 자신의 기준과 성장한 자신을 만날 수 있음**

## 5. 기준의 중요성 (06:27)

### 혼란스러운 세상

- 같은 근거를 가지고도 의견이 갈림
- 정보가 넘쳐나는 시대

### 명확한 기준의 힘

> **"불안감은 명확한 본인만의 기준에 따라 축소되기도 하더라고요"**

### 실패의 재정의

- 명확한 본인만의 테두리에서 선택
- 비록 틀린 선택이어도 **본인을 더 성장시키는 선택**
- 실패 경험이 나중 선택의 **성공 확률을 높여줌**

> **실패는 가슴 아프고 좌절스러운 경험이기보다는,  
> 좋은 선택과 기준을 가지기 위한 여정에서 만난 친구**

### 기준의 역할

> **"저의 기준이 저를 일으켜 세워줄 때가 많아요.  
> 이렇게 어지러운 세상을 정리할 수 있는 건 본인만의 기준밖에 없어요."**

## 핵심 요약

| 개념 | 설명 |
|------|------|
| **개발 철학** | 코드를 쓰는 본인만의 기준, 정답 없음 |
| **다른 사람의 시간** | 철학의 핵심 - 이해 시간 단축 |
| **틈을 주지 않기** | 물리적 거리 → 다른 내용이라는 착각 |
| **단언적 코드** | 확장 방지 → 빠르게 넘어갈 수 있음 |
| **실패 → 기준** | 실패를 통해 기준을 세우고 성장 |

## 코드 작성 원칙 (영상 기반)

| 원칙 | 이유 |
|------|------|
| **싱글톤으로 틈 제거** | 물리적 거리가 있으면 다른 내용으로 착각 |
| **객체 리터럴 선호** | 클래스보다 단번에 싱글톤임을 알 수 있음 |
| **if-else 2분법** | 인간의 뇌는 2분법을 좋아함, 건드릴 필요 없음 표현 |
| **중괄호 제거** | 확장하지 않겠다는 의지, 틈 제거 |
| **느낌표 제거** | 생각을 한번 더 해야 하는 부담 제거 |
| **의도적 중복** | 복잡성 감소 > DRY 원칙 |
