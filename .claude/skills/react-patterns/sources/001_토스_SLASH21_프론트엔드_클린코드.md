## 1. 실무에서 클린 코드의 의미 (01:00)

### 회사에서 한번쯤 들어본 말

> "아 그 코드는 안 건드리시는게 좋을 거에요. 일단 제가 만들게요"

**의미:**
- 흐름 파악이 어렵고
- 도메인 맥락 표현이 안 됨
- 동료에게 물어봐야 알 수 있는 코드

### 나쁜 코드의 결과

- 개발할 때 **병목**이 됨
- 유지보수할 때 **시간이 오래** 걸림
- 심하면 **기능 추가가 불가능**한 상태
- 성능도 안좋은 경우가 많음

### 클린 코드의 의의

> **유지보수 시간의 단축**

- 동료 혹은 과거의 내가 짠 코드를 **빠르게 이해**할 수 있다면 개발 시간이 짧아짐
- 코드 리뷰 시간 단축
- 버그가 났을 때 디버깅 시간 단축
- **시간은 자원이며, 자원은 곧 돈**

## 2. 기능 추가의 함정 (02:16)

### 처음 설계할 때 vs 기능 추가할 때

- 처음 설계하고 새로운 파일에서 짤 때는 잘 깨끗함
- 하지만 **기존 코드에 기능을 추가하는 상황**이라면 다름
- 긴장의 끈을 놓치면 코드가 엉망이 됨
- **회사에서 우리 일의 90%가 기존 코드에 기능을 추가하는 일**

### 나쁜 코드 추가의 문제점 (03:23)

1. **하나의 목적인 코드가 흩뿌려져 있음**
   - 관련 코드가 뚝뚝 떨어져 있어서 스크롤을 위아래로 이동하면서 미로 찾기를 해야 함

2. **하나의 함수가 여러 가지 일을 함**
   - 세부 구현을 모두 읽어야지 함수의 역할을 알 수 있게 됨
   - 코드 추가 및 삭제도 시간이 더 걸림

3. **함수의 세부 구현 단계가 제각각**
   - 비슷한 이름인데 하나는 여러 가지 일을 하고 있어서 읽을 때 어지러움
   - 코드를 이상하게 지레짐작하게 됨

### 그때는 맞고 지금은 틀리다

> 깔끔했던 코드가 작은 기능 하나를 추가했을 뿐인데 어지러운 코드가 됨

**함정:** PR에서의 변경 사항만 보면 어지러운 코드라는 것을 파악하기 어려움
- 변경점 자체는 틀린 점이 없으니까
- **하지만 전체 그림으로 보면 엉망**

## 3. 응집도 (07:51)

> **같은 목적인 코드는 뭉쳐두자**

### 안티패턴: 보기 더러울 때 일단 뭉쳐두기

커스텀 훅의 대표적인 안티패턴:
- 보기 더러우니까 일단 뭉쳐두는 것
- **고해성사: 저도 정말 많이 짠 패턴입니다**

### 무엇을 뭉쳐야 할까?

| 뭉쳐야 할 것 | 분리해야 할 것 |
|-------------|---------------|
| 당장 몰라도 되는 디테일 (세부 구현) | 코드 파악에 필수적인 핵심 정보 |

**핵심:** 뭉쳐서 짧은 코드로 만든다고 코드가 깨끗해지는 게 아님

> **클린 코드는 짧은 코드가 아닙니다. 찾고 싶은 로직을 빠르게 찾을 수 있는 코드입니다.**

### 핵심 데이터 vs 세부 구현 (10:03)

**핵심 데이터 (남겨야 할 것):**
- 팝업 버튼 클릭시 수행하는 액션
- 팝업의 제목
- 팝업의 내용

**세부 구현 (숨겨도 되는 것):**
- 팝업을 열고 닫을 때 사용하는 상태
- 컴포넌트의 세부 마크업
- 팝업의 버튼 클릭시 특정 함수를 호출해야 한다는 바인딩

### 선언적 프로그래밍 (10:03)

```tsx
// 선언적 스타일 - 무엇을 하는지 빠르게 이해 가능
<Popup
  title="보험 질문하기"
  content="전문가가 설명드려요"
  onConfirm={submitQuestion}
/>
```

**특징:**
- 무엇을 하는 것인지 빠르게 이해가 가능
- 세부 구현은 안쪽에 뭉쳐두어서 신경 쓸 필요가 없음
- 무엇만 바꿔서 쉽게 재사용할 수 있음

### 명령형 vs 선언적

- 선언적 프로그래밍이 명령형 프로그래밍보다 최신 트렌드니까 더 좋은 건가? **아니요**
- 두 방법 모두 의미가 있음
- 상황에 따라 명령형으로 개발해야 할 때도 있음

## 4. 단일 책임 (12:06)

> **하나의 일을 하는 뚜렷한 이름의 함수를 만들자**

### 함수 이름과 실제 동작의 불일치 (12:14)

```typescript
// ❌ 이름: 질문 제출, 실제: 약관 체크 + 질문 제출
function handleQuestionSubmit() {
  checkTermsAgreement(); // 이름에 없는 일
  submitQuestion();
}
```

**문제점:**
- 중요 포인트가 모두 담겨있지 않은 함수명
- 읽는 이가 예상한대로 코드가 동작하지 않음
- **코드에 대한 신뢰 하락**
- 그 다음부터는 함수명을 믿지 못하고 세부 구현을 모두 의심 가득한 눈초리로 보게 됨

### 기능 추가가 반복될 시 (13:22)

> "그 코드는 안 건드리시는게 좋을 거에요"

**해결책:** 한 가지 일만 하는 명확한 이름의 함수가 되도록 리팩토링

### 컴포넌트로 기능 분리 예시

```tsx
// 로그 찍는 기능을 컴포넌트로 분리
<LogClick>
  <Button onClick={handleApiCall}>클릭</Button>
</LogClick>
```

- 버튼 클릭하면 자동으로 클릭 로그가 전송됨
- 버튼 클릭 함수에서는 API 콜만 신경 쓸 수 있음

### 소소한 팁: 한국어 변수명 (14:31)

**유용한 상황:**
- 도메인이 복잡해서 영어 이름을 길게 짓는 게 오히려 복잡도를 높일 때
- 상수를 직관적으로 보고 싶을 때
- 복잡한 조건문이 많아질 때

```typescript
const 실시간상담가능 = isRealtimeConsultAvailable;
const 연결고객수5명미만 = connectedCustomerCount < 5;
```

> 마치 주석을 달아 놓은 것과 같은 효과. 그리고 코드가 약간 귀엽지 않나요?

## 5. 추상화 (14:31)

> **로직에서 핵심 개념을 뽑아내기**

### 피카소의 소 그림

- 왼쪽: 구체적으로 소를 그림
- 점점 디테일을 지워 나감
- 마지막: 본인이 소에서 남기고 싶은 개념만 남김 (몸통이 두껍고 뿔이 있네요)

### 지하철 노선도

- 현실 세계의 지하철 구조는 굉장히 복잡
- 지도 위에 색색의 선으로 추상화
- 더 간단하게 추상화: 이름과 호선만 남기고 모두 숨김

### 프론트엔드 코드도 동일 (15:37)

```tsx
// 구체적인 코드
<form onSubmit={handleSubmit}>
  <input value={question} onChange={...} />
  <button type="submit">제출</button>
  {isSuccess && <Toast message="성공" />}
</form>

// 추상화된 코드
<QuestionForm
  onSubmit={submitQuestion}
  onSuccess={showSuccessToast}
/>
```

### 추상화 수준 선택 (16:47)

구체적인 코드를:
- 조금 추상적이게
- 혹은 더욱 추상적이게

**정답은 없습니다. 상황에 따라 필요한 만큼만 추상화하시면 됩니다.**

### 추상화 수준이 섞여 있으면 (17:54)

> **한 레벨의 코드 안에 추상화 수준이 섞여 있으면 코드 파악이 어려워집니다**

```tsx
// ❌ 추상화 수준이 섞임
<Header />                              {/* 높은 추상화 */}
<div className="flex gap-4 p-8">        {/* 낮은 추상화 */}
  <span>{data.name}</span>              {/* 낮은 추상화 */}
</div>
<Footer />                              {/* 높은 추상화 */}

// ✅ 추상화 수준 통일
<Header />
<ProductDetail product={data} />
<Footer />
```

**문제:**
- 구체적으로 작성된 컴포넌트를 보고 다른 컴포넌트도 충분히 구체적으로 작성되어 있다고 짐작
- 내부를 보면 엄청나게 복잡한 코드가 숨겨져 있을 수도 있음
- 작은 코드라고 생각했는데 복잡한 코드가 나옴
- **코드를 읽는데 사고가 널뛰게 됨**

## 6. 액션 아이템 (18:58)

### 1. 담대하게 기존 코드 수정하기

- 구조 뜯기를 두려워하면 클린한 실무 코드를 유지할 수 없음
- PR에 파일 체인지가 많아진다는 명목하에 기존 코드를 건드리지 않고 추가만 하기 쉬움
- **두려워하지 마시고 기존 코드를 씹고 뜯고 맛보고 즐겨보세요**

**팁:** PR에 빨간색 체인지 수를 줄이고 싶다면 마더 브랜치를 따서 리팩토링 PR을 추가로 만들기

### 2. 큰 그림 보는 연습

- **그때는 맞고 지금은 틀릴 수 있다**
- 기존의 깨끗하던 코드에 내가 기능을 추가하면서 망쳐놓을 수 있음
- 한 기능 추가 자체는 클린해도 큰 그림으로는 어지러운 코드일 수 있음

### 3. 팀과 함께 공감대 형성

- 코드의 정답은 없음
- 사소하지만 일관성을 깨는 코드가 쌓이면 유지보수하기 힘든 코드가 됨
- **공감대는 자동으로 만들어지지 않음. 명시적으로 이야기하는 시간이 필요**

**방법:**
- 함께 만들어온 코드에서 고치고 싶은 포인트를 서로 말해보기
- 각자 문제라고 생각하는 지점을 공유해 집단지성을 모으기
- 어떻게 개선할 것인지 고민

### 4. 문서로 적어보기

- 클린 코드란 모호한 개념
- **글로 적어야 명확해짐**
- 이 코드가 어떤 점에서 향후 위험할 수 있는지
- 어떻게 개선할 수 있는지
- 나만의 원칙을 적어보기

> "저는 이 발표를 준비하면서 어지러운 머리 속을 정리할 수 있었어요"
